x402 is an open protocol that adds native payments to HTTP. It uses the `402 Payment Required` status code — defined in HTTP/1.1 but never widely adopted — to create a machine-readable payment flow.

**Spec:** [x402.org](https://x402.org)

## The flow

```
Client                              Server
  │                                    │
  │  GET /api/data                     │
  │ ──────────────────────────────────▶│
  │                                    │
  │  402 Payment Required              │
  │  {                                 │
  │    "accepts": [{                   │
  │      "scheme": "exact",            │
  │      "network": "eip155:8453",     │
  │      "price": "$0.001",            │
  │      "payTo": "0x..."              │
  │    }]                              │
  │  }                                 │
  │ ◀──────────────────────────────────│
  │                                    │
  │  [Client signs USDC payment]       │
  │                                    │
  │  GET /api/data                     │
  │  X-402-Payment: <base64-payload>   │
  │ ──────────────────────────────────▶│
  │                                    │
  │  [Server verifies via facilitator] │
  │                                    │
  │  200 OK                            │
  │  { "data": "..." }                 │
  │ ◀──────────────────────────────────│
```

### Step by step

1. **Request** — Client makes a normal HTTP request (no auth headers needed)
2. **402 Response** — Server returns `402 Payment Required` with an `accepts` array describing what payment it wants
3. **Sign** — Client reads the payment requirements, signs a USDC transfer matching the amount and destination
4. **Retry** — Client retries the same request with `X-402-Payment` header containing the signed payment
5. **Verify** — Server sends the payment to a facilitator for on-chain verification
6. **Response** — If payment is valid, server processes the request and returns the result

## Payment header

The `X-402-Payment` header contains a base64-encoded JSON payload:

```json
{
  "scheme": "exact",
  "network": "eip155:8453",
  "payload": {
    "signature": "0x...",
    "authorization": {
      "from": "0xBUYER_WALLET",
      "to": "0xSELLER_WALLET",
      "value": "1000",
      "validAfter": 1700000000,
      "validBefore": 1700003600,
      "nonce": "0x..."
    }
  }
}
```

## Facilitator

The facilitator is a server that verifies payments on-chain. AgentGate uses the facilitator at `x402.org/facilitator`.

The facilitator:
- Decodes the payment payload from the `X-402-Payment` header
- Checks the payment signature
- Verifies the USDC transfer matches the required amount and destination
- Returns a verification result to the server

The facilitator never touches the funds — it's a read-only verifier.

## Payment schemes

x402 supports different payment schemes. AgentGate uses the `exact` scheme:

- **`exact`** — Pay the exact price in USDC. One payment per request. No subscriptions, no prepaid balance. Uses `ExactEvmScheme` for Base and `ExactSvmScheme` for Solana.

## Why x402?

**For sellers:**
- No payment infrastructure to build
- No user accounts to manage
- No invoicing, no billing cycles
- Revenue per request, not per month

**For buyers (especially AI agents):**
- No signup flow to navigate
- No API key provisioning
- No rate limits to negotiate
- Pay exactly for what you use

**For the ecosystem:**
- Open standard, not a proprietary API
- Any HTTP client can participate
- Composable — chain multiple paid APIs together
- Permissionless — anyone can sell, anyone can buy
